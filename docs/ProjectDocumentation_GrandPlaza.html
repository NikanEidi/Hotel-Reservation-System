1. Project Overview


Summary

Grand Plaza Hotel is a five-star property located in Toronto, Ontario. The hotel previously managed all reservations manually using paper-based forms stored in filing cabinets. This system introduces a fully digital replacement covering the complete guest journey from initial inquiry through room selection, service add-ons, loyalty enrollment, payment, and checkout.


The system is a JavaFX desktop application consisting of two primary interfaces: a self-service kiosk intended for use by arriving guests, and a password-protected administration portal for hotel staff. Both interfaces communicate with a shared MySQL database through a JPA-based data access layer.


Key Features

- Self-service kiosk with a guided seven-step reservation workflow

- Room selection with four categories and occupancy validation

- Dynamic pricing using date-based strategy selection

- Add-on service selection using the Decorator design pattern

- Loyalty program enrollment and point tracking

- Booking summary with CSV and PDF export

- Admin dashboard with reservation search and reporting modules

- Secure BCrypt password authentication for admin accounts

- Rotating file-based audit logging with size limits

- JPA persistence for all entities with Hibernate as the ORM provider


Technologies Used

Language: Java 17

UI: JavaFX 17 with FXML declarative layouts

Database: MySQL 8

ORM: Hibernate 5.6 with JPA 2.2 specification

Security: jBCrypt 0.4 for password hashing

Export: OpenCSV 5.8, OpenPDF 1.3

Build: Apache Maven 3.8 via Maven Wrapper

Runtime: GraalVM CE 25 (Java 17 compatible)



2. Architecture Summary


Three-Tier Architecture

The application strictly separates concerns across three tiers.


The presentation tier consists of FXML layout files and their corresponding Java controller classes. Each screen in the kiosk and admin portal has its own FXML file and a dedicated controller. Controllers handle only UI events, validation messages, and scene transitions. No business logic resides in the controllers.


The business logic tier contains service classes that enforce all business rules including pricing calculations, occupancy checks, loyalty point accumulation, and authentication. This tier also defines the design patterns that structure the system's core behaviour.


The data tier consists of JPA entity classes mapped to MySQL tables, a singleton entity manager factory, and a utility class for database connection management. All database access goes through the EntityManager API.


Cross-Cutting Concerns

Logging is handled at every tier boundary. The LoggerUtil class configures a rotating file handler attached to the root logger. All significant actions including login attempts, reservation creation, errors, and database operations are captured in timestamped log entries.


Exception handling uses a layered approach. Service methods throw checked exceptions when business rules are violated. Controllers catch these exceptions and display appropriate error messages to the user without exposing internal stack traces.


MVC, DI, and ORM Usage

The Model-View-Controller pattern is applied throughout. FXML files are the views. Controller classes are the controllers. Entity classes and service objects are the models. UIController passes itself as a dependency to sub-controllers using a setMainController method, providing a lightweight manual dependency injection mechanism without a framework.


Hibernate ORM maps Java entity classes to relational tables. Annotations define column names, constraints, relationships, and cascade behaviour. JPQL queries in service classes retrieve filtered result sets from the database.

4. Entity and Relationship Mapping


Entities


Admin: Stores administrator credentials. Fields are id (auto-generated primary key), username (unique), password (BCrypt hash), and role.


Guest: Represents a registered hotel guest. Fields are id, name, email, and phone. The phone number serves as the lookup key for loyalty status.


Room: Abstract base entity mapped using InheritanceType.JOINED. Subclasses are SingleRoom, DoubleRoom, and PenthouseRoom. Fields are id, roomType (enum string), and pricePerNight.


Reservation: Represents a booking. Fields are id, checkIn and checkOut dates, numberOfGuests, status, and a reference to the associated Guest. It also holds a list of Room objects.


Payment: Represents a completed or pending payment. Fields are id, amount, paymentMethod, and paymentDate. It references a Reservation.


Relationships and Annotations

Guest to Reservation: One-to-many. A guest can have multiple reservations over time. The Reservation entity holds the foreign key.


Reservation to Room: Many-to-many. A reservation can include multiple rooms. A room can be included in multiple reservations over different date ranges. This relationship is mapped using a join table.


Payment to Reservation: One-to-one. Each payment record corresponds to exactly one reservation.


Cascade and Fetch Notes

Reservation cascades persist and merge operations to its associated Guest. Fetch type for Room lists is LAZY to avoid loading all room data on every reservation query. Payment uses EAGER fetch of its associated Reservation for billing display.



5. Design Pattern Usage


Singleton - JPAUtil

The EntityManagerFactory is expensive to create and should exist only once per application lifecycle. JPAUtil uses a static field and a synchronized get method to ensure a single EntityManagerFactory is shared across all service classes. This is the classic thread-safe lazy singleton pattern.


Factory - RoomFactory

RoomFactory provides a static createRoom(String type) method that constructs and returns the appropriate Room subclass (SingleRoom, DoubleRoom, or PenthouseRoom) based on the type string. This isolates object creation from business logic and makes it easy to add new room categories without changing existing code.


Strategy - Pricing

Three pricing strategies implement the PricingStrategy interface: StandardPricing, WeekendPricing (multiplier 1.2), and PeakSeasonPricing (multiplier 1.3). UIController.getPricingStrategy() inspects the selected dates and returns the appropriate strategy at runtime. The strategy's calculatePrice() method is then called to determine the final room cost. This makes it trivial to add new pricing rules (e.g., holiday surcharges) without altering the pricing calculation entry point.


Decorator - Add-on Billing

The BillComponent interface defines getCost() and getDescription() methods. BaseBill holds the base room cost. Each add-on service (WiFiDecorator, BreakfastDecorator, ShuttleDecorator, SpaDecorator) wraps an existing BillComponent and adds the cost of one service on top. The decorators can be stacked in any order, producing a final bill that reflects all selected services. This eliminates conditional branching for pricing and makes adding a new service a matter of writing one new decorator class.


Observer - Waitlist

The Observer pattern is planned for the waitlist notification system. When a room type becomes available due to a cancellation, a RoomAvailabilityEvent is fired. All registered WaitlistObserver instances receive the notification and can display an alert to the waiting guest. This pattern decouples the availability check from the notification delivery mechanism.



6. Business Rules


Occupancy Limits

Single Room: maximum 2 guests

Double Room: maximum 4 guests

Deluxe Room: maximum 2 guests

Penthouse Suite: maximum 2 guests


Total capacity is calculated as the per-room maximum multiplied by the quantity selected. If the guest count entered on the search screen exceeds total capacity, the room selection screen blocks navigation and displays the exact difference.


Pricing Rules

Base rates per night:

Single Room: 89 CAD

Double Room: 129 CAD

Deluxe Room: 199 CAD

Penthouse Suite: 349 CAD


Weekend surcharge: 20 percent markup applied when the majority of nights in the stay fall on a Saturday or Sunday.

Peak season surcharge: 30 percent markup applied when the stay overlaps with July, August, or December.


Room base rate multiplied by the applicable strategy factor gives the per-night price. That value is multiplied by room quantity and number of nights.


Add-on Pricing

High-Speed WiFi: 9.99 per device per night

Daily Breakfast Package: 24.99 per person per night

Airport Shuttle: 35.00 per trip (flat fee)

Spa and Wellness: 49.99 per session (flat fee)


Tax

13 percent HST is applied to the combined room and add-on subtotal. This matches the Ontario provincial tax rate applicable to hotel accommodations.


Loyalty Program Rules

Enrollment is offered at the loyalty step during every booking. Guests earn 1 point for every 10 dollars spent on their stay. A balance of 100 points can be redeemed for a 10 dollar discount on a future booking. Upgrades and exclusive member rates are offered subject to availability.


Cancellation Policy

Cancellations must be made at least 24 hours before the scheduled check-in time. Late cancellations or no-shows are subject to a one-night penalty charge.



7. Security and Logging


Authentication

Admin passwords are never stored in plain text. When a new admin is registered, the password is passed through BCrypt's hash function before being written to the database. During login, the plain-text entry is compared against the stored hash using BCrypt.checkpw(). The Admin entity holds only the hash.


The initial admin account (username: nikan) is seeded by the Launcher on first run. If the account already exists, the duplicate entry exception is silently caught and ignored.


Logging Configuration

LoggerUtil configures a java.util.logging.FileHandler with:

Log file pattern: system_logs.%g.log

Maximum file size: 1 MB per file

Maximum retained files: 10 (rotating)

Formatter: SimpleFormatter with timestamp, level, and message


Sample log entry format:

Apr 07, 2026 12:34:51 AM com.seneca.hotelreservation_system.service.AuthService authenticate

INFO: Admin login attempt for user: nikan - Result: SUCCESS


Exception Handling Strategy

Service methods declare checked exceptions for recoverable errors (invalid credentials, constraint violations, not-found entities). Controllers wrap service calls in try-catch blocks and display user-friendly Alert dialogs. Unrecoverable errors (database connection failure at startup) cause Launcher to print to System.err and exit before the UI is shown.



8. Export and Reporting


CSV Export

From the booking summary screen, guests can export their reservation details to a CSV file. The file is structured in labelled sections: Guest Information, Stay Details, Room Details, Add-ons, and Price Breakdown. Each field is written as a label-value pair on its own row. The file is saved to a user-selected location using a standard file chooser dialog.


PDF Export

The PDF export produces a plain-text receipt formatted with separators and column-aligned values. The header reads GRAND PLAZA HOTEL with the address and establishment year. Sections mirror the CSV structure. The footer thanks the guest and confirms that payment will be collected at the front desk.


Both export formats use the OpenPDF and OpenCSV libraries included as Maven dependencies.


Admin Reports

The admin dashboard provides a reservation table that can be filtered by guest name, phone number, or reservation ID. Revenue and occupancy summary cards appear at the top of the dashboard. Extended reporting modules for billing, feedback, and waitlist management are accessible from the sidebar navigation.



9. Challenges and Learnings


Technical Challenges


The most significant architectural challenge was managing shared state across seven sequential kiosk screens without a framework. JavaFX does not natively support a shared session object between independently loaded FXML controllers. We resolved this by having UIController hold the BookingData inner class and passing itself to each sub-controller via a setMainController method. This was a deliberate compromise between simplicity and strict separation of concerns.


ToggleGroup elements in FXML caused a runtime crash early in development. JavaFX attempts to add ToggleGroup as a Node child of its parent container, but ToggleGroup is not a Node. The solution is to declare it inside the parent element's resources block, then reference it from RadioButton elements using the dollar sign prefix syntax.


Hibernate's schema generation on a fresh database required careful management of entity annotations. Missing cascade annotations caused detached entity exceptions during reservation persistence. Understanding which operations should cascade and which should not required iterative testing against a live MySQL instance.


Individual Reflections


Working on this project taught the value of establishing a clear data flow early. We spent significant time late in development refactoring because early controller classes were too tightly coupled. A more deliberate upfront design of the BookingData session object would have saved considerable rework.


The decorator pattern for add-on billing was initially unfamiliar but proved to be one of the most elegant solutions in the codebase. Once the base interface was established, each new add-on type took only minutes to implement without touching existing code.


JavaFX's approach to scene management differs significantly from web-based frameworks. Loading a new FXML file replaces the entire scene graph, requiring that all stateful data be serialized to the session object before navigation. This constraint reinforced disciplined state management throughout the kiosk flow.


Suggestions for Improvement


A dedicated session manager class would replace the current UIController-as-hub approach. Each controller would inject the session manager rather than receiving a reference to UIController, reducing coupling and improving testability.


Unit tests for the service layer would have caught the pricing calculation bugs earlier. Adding JUnit 5 tests for PricingStrategy, the Decorator chain, and occupancy validation would increase confidence during refactoring.


The current persistence layer uses Hibernate's built-in connection pool which is not intended for production. A connection pool such as HikariCP should replace it before any real deployment.


The admin dashboard statistics (total reservations, rooms occupied, pending check-ins, waitlist) currently show placeholder values. Connecting these labels to live JPQL aggregate queries in BookingService would complete the reporting feature.
